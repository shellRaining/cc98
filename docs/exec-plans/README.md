# 执行计划

执行计划记录一次复杂改动准备怎么推进，以及实施过程中发生了什么。它是工作中的活文档，不要求在开工时预见所有细节。

## 目录约定

- `active/`：正在调研、实施或仍有明确交付物的计划。
- `completed/`：已经验收的计划和历史迁移记录。
- `template.md`：新建复杂计划时使用的起点。

计划完成后，先补齐结果、遗留项和验证记录，再从 `active/` 移入 `completed/`。归档文件只在追溯背景或复用决策时读取，不进入默认开工上下文。

## 完成与归档

执行计划在目标交付物已经落地、必要验证已经通过、没有仍需当前任务完成的明确步骤时视为完成。状态写成“已实施”“调研完成”或勾完大部分清单都不能代替这次核对。

开放式观察、可选优化和适合独立推进的后续工作不必让计划长期留在 `active/`。把它们写入遗留项，必要时新建计划或 issue，再归档已经验收的当前计划。仍有具体交付物、阻塞中的必要验证或尚未兑现的目标时，计划继续留在 `active/`，并更新状态、进展和下一步。

关联计划的任务结束前，agent 必须完成一次收尾检查：

1. 对照目标、实施步骤和验证章节，确认哪些内容已经完成。
2. 补齐结果、验证命令与结论、范围变化和遗留项，勾选已经完成的步骤。
3. 完成时把文件移入 `completed/`，同步本页两张索引表及仓库内相关链接。
4. 未完成时保留在 `active/`，更新状态和下一步，最终回复不得把任务描述为已经全部完成。

每日 Librarian 会补查主分支提交造成的文档漂移和遗漏归档，但它只负责兜底，不替代实施任务自身的收尾检查。

## 何时写

适合写执行计划的情况：

- 改动跨越多个模块或需要分阶段落地
- 实现前有重要技术问题需要调研或试验
- 工作包含迁移、兼容、回滚等风险
- 任务可能跨越多次会话，需要给后续 agent 留下上下文

小范围修复、机械调整和路径明确的单文件改动通常不用写。

## 怎么写

文件名使用 `YYYY-MM-DD-kebab-case.md`，新计划放入 `active/`。可以复制 `template.md` 开始，也可以根据任务重新组织结构。

模板只提供常见内容的提示，不是必须填满的表单。保留对当前任务有用的章节，删掉无关内容；需要时可以加入调研结论、原型、接口草图、迁移清单或其他更合适的表达。涉及架构图时使用 Mermaid。

计划应随着实施更新：完成阶段、范围变化、验证结果和未解决问题都写回原文。状态变化时同步更新本页；执行结束后勾选已经完成的步骤，记录结果与遗留项，再把文件移入 `completed/`。

## 与 ADR 的边界

执行计划回答“这次改动怎么做”，可以包含暂定方案和仍待验证的判断。

ADR 回答“长期为什么这样设计”。实施中确认了会长期影响架构边界、核心依赖或设计范式的决策，再单独写入 `docs/adr/`，不要把整份执行计划转成 ADR。
