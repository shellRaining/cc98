# 网站 CSS 所有权迁移

> 状态：已完成。业务样式已经迁回对应 SFC，并通过匿名态、登录态、主题和响应式浏览器对照。

## 背景

高保真迁移前，`apps/website/src/styles/global.css` 约 165 行，主要存放主题 token、基础页面样式和全局 reset。高保真迁移完成后，该文件增长到约 5700 行，Header、首页、版面、主题、用户中心、消息和管理页的样式都集中在同一个文件中。

这些样式已经按 BEM 前缀区分功能，但文件位置没有反映组件所有权。修改一个组件时，需要在 Vue 文件和长篇全局 CSS 之间来回查找；删除组件或调整结构后，也难以判断对应选择器是否仍有消费方。文件末尾的响应式规则还混合了多个功能域，使局部调整容易影响无关页面。

本次迁移只调整样式的文件归属。现有选择器、声明、断点、布局和交互效果保持不变，视觉修正、类名整理和样式复用留到后续改动。

## 目标

- 页面和组件样式放回渲染对应 DOM 的 Vue SFC，样式与模板可以一起阅读、修改和删除。
- `global.css` 最终只保留主题 token、文档级基础样式、reset 和真正跨组件的全局原语。
- 每条业务选择器都有明确所有者，跨组件选择器和响应式规则不再集中在全局文件末尾。
- 迁移前后的页面结构、尺寸、颜色、交互状态和主题表现一致。
- 每个功能切片可以独立验证和回退，不依赖一次性改完整个文件。
- 所有权迁移验收后，整理一份 UnoCSS utility 候选清单，供后续重构评审。

## 非目标

- 不改视觉设计，不调整间距、字号、颜色、圆角、阴影或响应式布局。
- 不重命名 class，不改 BEM 结构，不在所有权迁移中把现有 CSS 改写成 UnoCSS utility。
- 不合并重复声明，不清理 `!important`，不顺手修复迁移中发现的样式问题。
- 不调整 Vue 模板结构、组件拆分、路由、状态管理和数据请求。
- 不迁移 `apps/website/src/lib/`。领域逻辑的目录收敛另开计划。
- 不修改 `skins.css` 的皮肤变量组织，除非实施中发现某条规则并非皮肤变量而是业务样式。

实施中发现的视觉问题和无效选择器只记录，不在本计划内处理。所有权迁移完成后会审计适合改写成 UnoCSS utility 的规则，本计划只产出候选清单，不实施转换。只有阻止样式安全迁移的问题才随当前切片修复，并在进展记录中说明原因。

## 所有权规则

样式按下面的顺序判断所有者：

1. 选择器匹配的元素由某个 SFC 直接渲染时，样式归该 SFC。
2. 多个页面通过同一个业务组件渲染相同结构时，样式归业务组件，不归调用它的 View。
3. View 或 Layout 负责多个子组件之间的网格、间距和页面级排列时，只保留控制这些根节点关系的布局样式。
4. 只有文档根节点、全站基础元素和跨功能稳定原语可以留在 `global.css`。

复用不是本轮抽象目标。迁移时不因为两段声明相似就新建公共 class 或共享样式文件。已有多个消费方的同一结构应先确认是否已经有共同组件；没有共同组件时保持现状，后续再决定是否抽象。

### Scoped 样式

- 只匹配当前 SFC 模板节点的规则移入 `<style scoped>`。
- 选择器涉及子组件内部节点时，优先把对应规则继续下移到子组件，不用父组件依赖子组件的内部结构。
- 父组件确实负责插槽内容、富内容后代或第三方组件外观时，可以使用 `:deep()`，但需要保留原选择器的作用范围。
- 页面级状态 class 由父级控制、视觉落在子组件时，应确认状态属于父级布局还是子组件变体，再决定使用 `:deep()` 或把状态作为显式 prop/class 传入。本轮不为了消除 `:deep()` 改组件 API。
- 无法确认 scoped 编译后优先级是否一致的规则，先完成浏览器对照再移除全局副本，不能凭静态判断直接删除。

Vue scoped CSS 会给选择器增加属性选择器，可能改变优先级。实施时需要检查与全局链接样式、UnoCSS utility、组件状态 class 和第三方样式之间的覆盖关系。不能把“声明没有变化”当作行为必然不变。

### UnoCSS 与 SFC CSS 的边界

所有权迁移保持现有 CSS 写法。规则移入 SFC 时仍使用原选择器和声明，不同步改成模板中的 UnoCSS utility，也不新增 shortcut。模板里已经存在的 UnoCSS class 保持不变。

长期边界按样式性质判断：

- 语义颜色、间距和圆角继续从 UnoCSS 映射的设计 token 取值。
- Button、Input、Badge 等跨业务基础组件继续由 `components/ui`、cva 和 UnoCSS 管理。
- 单个元素上的简单静态布局和排版可以列为 UnoCSS utility 候选。
- 组件专属的复杂网格、后代关系、伪元素、状态组合和响应式规则保留在 scoped CSS。
- 多处声明相似不等于需要新增 shortcut。只有存在稳定复用语义时，才考虑抽成基础组件或 UnoCSS shortcut。

候选审计在全部样式完成所有权迁移并通过浏览器验收后进行。审计只记录候选位置、建议写法、复用范围和风险，转换实现放到独立提交或后续执行计划中。

### 规则移动

- 复制规则时保持选择器、声明和值不变，确认新位置生效后再从 `global.css` 删除原规则。
- 同一所有者的规则保持原有相对顺序。存在依赖源码顺序的覆盖时，先记录覆盖关系，再决定放入同一个 style block 还是保留明确的后置规则。
- `:hover`、`:focus-visible`、`:visited`、`:disabled`、伪元素和状态修饰符必须和基础规则一起迁移。
- `@media` 中的规则拆回各自所有者，并放在对应 SFC 的基础样式之后。断点和值保持不变。
- 不在多个位置长期保留同一规则。短暂复制只用于当前切片验证，切片完成前必须删除旧规则。

## `global.css` 保留范围

计划完成后，`global.css` 只保留下列内容：

- `:root` 下的原始调色板、语义 token 和尺度 token。
- `data-theme` 对应的全局明暗模式变量。
- `html`、`body`、`#app` 的页面基础样式。
- `box-sizing` reset 和全站基础链接行为。
- `.cc98-content` 这类明确跨页面复用的版心原语。
- 经审查确认必须跨组件生效的少量全局规则，并在计划进展中记录保留原因。

`site-header`、`home`、`board`、`topic`、`user`、`message`、`writing`、`search`、`focus`、`recommended`、`site-manage` 等业务前缀不应继续留在 `global.css`。

## 验证

每个阶段完成后运行受影响范围的 `vp check` 和相关 Vitest。样式迁移主要依赖浏览器验证，不能只用静态检查判断完成。

浏览器验证至少覆盖：

- 1440×1000 和 1140px 宽度。
- 夏季亮色和默认暗色，最终验收增加一套节日皮肤。
- 首页、版面列表、版面、主题、写作、用户中心、消息和管理页面。
- 默认、悬停、键盘焦点、禁用、展开、下拉和弹窗状态。
- 长标题、长正文、空数据、匿名 401 和图片加载失败等现有边界状态。

动态内容不要求整页像素完全相同，但以下项目必须一致：

- 版心宽度、网格列宽、Header 和横幅高度。
- 字号、行高、间距、边框、颜色、背景图和元素显隐。
- 下拉菜单、Dialog、悬停、焦点、访问链接和禁用状态。
- 主题切换后的语义 token 消费结果。
- 控制台没有新增 Vue、CSS、资源加载或可访问性错误。

最终运行 `vp run ready`。如果某个切片无法在不改变行为的前提下迁移，保留原规则并在计划中记录阻碍，不用视觉近似替代现有实现。

## 风险

- Vue scoped CSS 会提高选择器优先级，可能覆盖 UnoCSS utility 或全局链接状态。
- 父组件样式可能依赖子组件内部 DOM，直接加 `scoped` 后会失效。
- Vite 按组件加载样式后，源码顺序可能与当前单文件顺序不同，依赖后置覆盖的规则需要单独处理。
- 文件末尾的响应式规则混合多个功能域，漏迁后只会在特定视口出现回归。
- 登录态、权限页和动态数据不容易稳定截图，需要同时核对结构和计算样式。
- 当前 CSS 中可能已有缺陷。本轮以迁移前行为为基线，不把旧问题误判为迁移回归，也不在迁移提交中顺手修复。
