# 图片资源按所有权重新组织

## 背景

`apps/website/public/` 目前有 138 个文件，约 18 MB。目录同时包含首页和登录页插图、错误状态图、年度总结背景、版面图标、头像框、皮肤横幅、品牌图标和 favicon。部分文件按页面分组，部分沿用旧站的 `/static/images` 路径，皮肤资源又使用单独的 `/skins` 路径。代码既有直接字符串 URL，也有 CSS `url()` 和按版面名称拼接的动态路径，无法从目录判断资源属于哪个功能、是否必须保留固定 URL。

旧前端源码位于 `/Users/shellraining/Documents/Forum`，上游为 `ZJU-CC98/Forum`。旧项目把部署资源放在 `dist/static/images/`，按表情、麻将、头像框、主题横幅等类型建立子目录，再由 `copy-webpack-plugin` 整体复制。这个结构适合旧站大量运行时拼接 URL 的方式，也给当前迁移提供了资源分类和兼容路径依据。它同时混合源码与构建产物，根目录仍有 117 个散落文件，总资源约 57 MB，不适合直接复制到 Vite 项目。

当前项目可以参考旧站按资源族分目录的做法，但需要重新划分 `src/assets` 与 `public` 的职责。Vite 能处理的静态资源应进入模块依赖图，只有外部契约要求固定 URL 的文件留在 `public`。

## 目标

- 每张图片都有明确的业务所有者，目录名称能够说明用途。
- 静态引用进入 Vite 依赖图，删除或移动文件时由类型检查或构建暴露断链。
- `public` 只保留固定 URL 契约和站点根文件，不再作为通用图片目录。
- 版面图标、头像框和皮肤这类资源族有单一注册入口，动态加载范围可审计。
- 保持现有页面、皮肤编号、头像行为、版面图标回退和对外 URL 兼容，不改变视觉效果。

## 非目标

- 不在本轮统一转换 WebP、AVIF 或 SVG，也不调整图片尺寸和压缩质量。
- 不重新设计插图、横幅、头像框和版面图标。
- 不迁移旧站尚未接入的表情、麻将和编辑器素材。
- 不改变 CDN、缓存头或部署平台配置。
- 不以内容哈希相同为由删除不同业务名称的图片。多套皮肤或多个版面可以有相同文件内容，但仍表达不同配置项。

## 调研结论

旧项目有三点可以继续沿用：资源族使用独立目录；运行时按编号或名称查找的图片保持稳定命名；主题横幅、卡片图和缩略图作为一组管理。

旧项目的 `dist/static/images` 不能作为当前目标结构。`dist` 同时承担源码库存和构建输出，Webpack 将整个目录复制，未使用文件不会被构建发现。大量文件直接放在图片根目录，功能代码依赖字符串路径，移动资源只能靠全文搜索和人工检查。

当前项目还存在两个兼容边界：

- `UserSettingsView` 会把 `/static/images/default_avatar_boy.png` 作为头像 URL 提交给服务端，这个地址不能改成带构建指纹的 URL。
- 版面图标目前根据 API 返回的版面名称生成 `/_<版面名>.png` 路径。它们没有逐文件文本引用，但属于一个完整的动态资源族。

实施前需要再检查真实 API 返回的头像、版面 Logo 和富内容中是否存在站内 `/static/images/...` 地址。确认被服务端数据或历史内容引用的路径后，将其列为兼容资源，不随普通静态图片迁移。

## 资源边界

### 进入 `src/assets` 的资源

满足以下任一条件的图片进入模块依赖图：

- 由 Vue、TypeScript 或 CSS 中的固定位置使用。
- 只属于一个页面或组件，没有对外稳定 URL 契约。
- 属于可枚举的资源族，可以通过静态 import 或 `import.meta.glob` 建立注册表。

目标目录按业务域组织：

```text
apps/website/src/assets/
├── annual-review/
├── board-icons/
├── brand/
├── home/
├── login/
├── discovery/
├── status/
├── themes/
│   └── <skin-id>/
└── user/
    ├── anonymous/
    └── avatar-frames/
```

目录使用英文 kebab-case，文件使用用途名称，不保留 `static/images` 这类部署结构。中文业务名称只在确实参与运行时映射的资源中保留，例如版面名称和头像框名称。

### 留在 `public` 的资源

`public` 只保留两类文件：

- 浏览器或站点根目录约定的文件，例如 `favicon.ico`。
- API、用户数据、历史富内容或其他外部消费者要求固定 URL 的兼容资源。

预期最小结构如下，最终以兼容性审计结果为准：

```text
apps/website/public/
├── favicon.ico
└── static/images/
    └── default_avatar_boy.png
```

新增 `public` 图片时必须说明为什么不能通过模块 import 使用。

## 各资源族的处理

| 当前资源                 | 目标位置                                      | 引用方式                                 |
| ------------------------ | --------------------------------------------- | ---------------------------------------- |
| `home/*`                 | `src/assets/home/`                            | 组件静态 import                          |
| `login/*`                | `src/assets/login/`                           | 登录页静态 import                        |
| 错误页与搜索空态         | `src/assets/status/`、`src/assets/discovery/` | 配置模块静态 import                      |
| `annual-review/*`        | `src/assets/annual-review/`                   | Vue 与 CSS 模块引用                      |
| `98LOGO.ico` 等品牌资源  | `src/assets/brand/`                           | Header 静态 import                       |
| `相框/*`                 | `src/assets/user/avatar-frames/`              | `avatar-frame.ts` 静态注册表             |
| 匿名头像和本地回退图     | `src/assets/user/anonymous/`                  | 共享头像模块导出 URL                     |
| `_<版面名>.png`          | `src/assets/board-icons/`                     | `import.meta.glob` 建立名称到 URL 的映射 |
| `skins/*`                | `src/assets/themes/<skin-id>/`                | CSS 相对 URL，主题预览使用静态 import    |
| `default_avatar_boy.png` | 暂留原固定 URL                                | 服务端持久化兼容                         |
| `favicon.ico`            | 保留 `public/` 根目录                         | HTML 固定 URL                            |

版面图标由 `components/board/` 持有加载规则。注册表从文件名生成版面名称映射，查不到时返回 CC98 默认图标，不在业务页面拼接文件路径。头像默认图、匿名头像和头像框也由 `components/user/` 导出语义化资源，不让页面重复硬编码 URL。

皮肤资源继续按 `skin-id` 成组存放，每组包含 `banner`、`banner-card` 和可选的暗色变体。`stores/skins.ts` 与 `styles/skins.css` 使用同一组目录名称，构建必须能发现缺失文件。图片路径整理不合并皮肤配置，也不改变 30 个旧编号到现有皮肤 ID 的映射。

## 验证

自动检查包括格式、lint、类型、Vitest、Knip 和生产构建。资源专项测试至少覆盖：版面名称映射与回退、头像框注册表完整性、皮肤配置文件存在性，以及服务端默认头像固定 URL 不变。

构建产物检查需要确认两件事：`src/assets` 图片由 Vite 输出为带指纹的文件，源码中没有残留旧路径；`public` 兼容资源仍可通过原 URL 访问，服务端保存的默认头像地址不受版本发布影响。

## 风险与回滚

- 历史帖子或 API 数据可能保存站内绝对路径。兼容性审计未完成前不删除对应 `public` 文件。
- CSS 中缺失的根路径资源不一定在类型检查阶段报错，皮肤迁移必须经过生产构建和浏览器网络检查。
- 版面名称来自服务端，文件名归一化不能改变空格、间隔号和大小写语义。映射规则保持精确匹配，只在查找失败时使用默认图标。
- 迁移按资源族分批完成。每一批同时移动文件和更新消费者，不设置长期双份资源。出现回归时可以按批次恢复原路径，不影响其他资源族。

## 决策记录

- 旧项目只作为资源分类和兼容路径的事实来源，不复制其 `dist/static` 整目录方案。
- 资源按业务所有权组织，不建立新的通用 `images` 目录。
- 动态资源使用显式注册表或静态 glob 进入构建依赖图，不再依赖任意字符串拼接。
- 图片格式和体积优化单独处理，避免路径迁移与视觉质量变化混在同一轮验证中。
