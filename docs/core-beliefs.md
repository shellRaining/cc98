# Agent-first 操作铁律

核心约束，参考 OpenAI Harness Engineering 整理。

## AGENTS.md 是目录，不是百科

控制在约 60 行，只放索引，深度内容放 `docs/` 下按需加载。context 是稀缺资源，巨型指令文件会挤掉任务本身和相关代码。

## 仓库知识是唯一真相源

agent 运行时看不到的东西等于不存在。GitHub 讨论、聊天记录、人脑里的约定，都得编码进仓库（markdown / schema / 代码）才能被 agent 用到。

## 按需展开，不要一次灌满

agent 从小而稳定的入口出发，需要时再往下读。文档分层级，避免一次性塞太多。

## 边界处 parse data shapes

数据进系统的地方必须验形状（Zod / valibot / 原生校验都行，不指定库）。内部依赖已验证的类型，不重复校验。

## 文档和代码同源

改公共 API 必须在同一次改动里更新 `docs/`。CI 和 pre-commit 会校验断链和结构。文档过期，agent 就照着错的做。

## 管住不变量，不卡实现细节

用 lint / 结构测试 / 类型系统管住架构边界和命名约定，实现细节不卡 agent。输出正确、可维护、对未来 agent run 可读，就算达标。

## 复杂改动先写执行计划

复杂工作先在 `docs/exec-plans/`（如启用）或本地执行计划里写清进度和决策日志，不在 chat 里堆长上下文。计划本身是版本化产物。

## 技术债持续还，不要攒着

技术债像高息贷款，持续小额还比攒着一次性清算划算。质量分和技术债追踪定期刷新。
